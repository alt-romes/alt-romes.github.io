  • More insight gained from reading Call-by-name, call-by-value, call-by-need and the linear lambda calculus:

The call-by-name calculus is not entirely suitable for reasoning about functional programs in lazy languages, because the beta rule may copy the argument of a function any number of times. The call-by-need calculus uses a diferent notion of reduction, observationally equivalent to the call-by-name calculus. But call-by-need, like call-by-value, guarantees that the argument to a function is not copied before it is reduced to a value.

This is quite relevant to the reverse-binder-swap transformation we discuss in the thesis, because if we allow the call-by-name reduction and reverse binder swap, we can effectively duplicate linear resources. If we disallow call-by-name reduction for linear applications (which seems quite useful), we could have the reverse-binder-swap with the added CaseVar rule. This paper agrees on how call-by-name (unsuitably to linear types) copies arguments.

That paper still looks really really relevant, the way they discuss duplication of resources in call-by-need vs call-by-name vs call-by-value.

As an application of these results, we have devised a type system that can infer information about which variables are used linearly in a call-by-need or call-by- value lambda calculus. Such types are useful for program transformation: the reduc- tion (( x: M )N ) → M [x := N ] does not in general hold for a call-by-value or call- by-need calculus, but it does hold if x is used linearly. Once upon a Type, the paper referred above, presents a variable usage inference algorithm that can help determine which variables are used once, to do * Safe and always efficient inlining * Avoiding closure update * Enabling data structure update

Since our system can see more linearity, we’d be able to inline expressions more often; effectively resulting in a smarter inliner.

The way GHC determines whether to inline arguments of function applications from usages seems to be based on “Once upon a Type”, as they claim a sound inlining reduction in a call-by-need setting through counting usages in the types. If indeed this is the strategy used for reduction, then reverse binder swap is safe; where-as if our system is used, reverse-binder-swap isn’t possible (since we either accept the transformation and duplicate resources (bad!), or reject the transformation).
